GET和POST的不完全对比

发表于 2018-07-03

​ 关于GET和POST是平常使用非常多的向服务端请求的方法，POST和GET也有着很多的相同或者不同点，一直没有非常清晰的捋一捋这二者的关系，现在重新来回顾，学习一下GET和POST的相关知识。

  1. 语义出发

    ​ 从语义上看，GET，获取的意思。POST，发送的意思。这一点很重要，在设计RESTful接口的时候，个人理解就是合理的运用请求方法的正确语义与功能实现。虽说语义上是这样，最好来按照这种方式来实行，但也不是一定的，也可以不遵守，比如说，ajax,请求的方法就可以是GET或者的POST，依据具体的场景考虑。表单的提交，既可以使用POST，和可以使用GET，当然，这是一种可以说错误，危险的行为，但是证明了一点。他们可以做到请求或者是发送。

  2. 是否可以缓存

    ​ GET请求可以缓存，大家都说GET请求是不安全的，POST请求是安全的，其实从是否可缓存这一点上面可以看出一点端倪。通俗的说，不可缓存的POST要比GET请求安全一点点，当然，不仅仅是体现在可缓存这一点上，还体现在别的方面，稍后再说。

  3. 在http协议的存在位置

    ​ 对于GET请求，一般是会将请求信息放在浏览器的url中进行传输，这不是绝对的，url是放在request header中的 。POST请求的请求信息是存在于 request body

  4. 安全性的体现

    ​ 因为GET请求会将请求的数据待在URL中，所以是相当于明文传输，自然没有安全性可言，POST是将数据放在了request body中，相比之下比较安全，但是，在http 1.1 中,可以利用抓包，然后还是可以获取到请求的信息，也是不安全的，这里的安全是相对于GET安全一点。其实这是http协议的一个痛点，所以，国内外，很多的大公司，极力推行所有的网格使用https协议，简单的将就是加密后进行传输。更安全。

  5. 大小限制——包含错误认识

    ​ 对于GET请求（请求信息放在url中，url 是放在 request header）。这里有一个非常大的误区，GET请求的长度有限制？其实HTPP协议中没有提出限制url的长度，我们常说的url的长度限制是浏览器对url请求长度的限制，每个浏览器不尽相同，总体来说，一般不超过2K不会出现比较大的问题。POST请求，HTTP协议也没有限制大小，限制大小的是各服务器厂商。

  6. 点击回退/刷新

    ​ GET 请求对此不做任何的处理，一般也称无害，对于POST请求，后退或者刷新，会重新提交(但是一般情况，浏览器应该告知用户将会被重新提交，我记得比较清楚的是我们学校的OJ平台，在进行后退，或者刷新后，会有这样的一个弹窗，告知将会重新提交表单)

​ 关于GET和POST请求，暂时就先说这么多，等有了更新的认识后再来补充。

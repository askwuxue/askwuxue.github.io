Http协议相关内容

发表于 2018-09-15 | 分类于 网络协议

http 协议是一个非常重要的方面，学习了HTTP协议，觉得HTTP不仅仅对于后端非常重要，对于前端也是非常重要，如果想要成为真正的开发者或者是全栈工程师，HTTP协议必须要能够精通。最近学习了HTTP协议，正好写一篇笔记，巩固一下相关内容，为后续的学习，打下一个良好的基础。

一道经典的面试题

有一道经典的面试题，从输入URL，到按下回车，页面显示。中间经历了什么？这个问题真的是非常开发的一个问题，可以答得内容非常多，最近看了HTTP协议，从网络的层面重新回顾一下这个问题。

三次握手的过程

http协议是建立于TCP连接之上的，如果要进行HTTP请求和相应，必须要建立TCP，建立TCP连接需要三次握手的过程，是为了防止网络发送失败导致的资源消耗与浪费。三次握手，通信双方可以确定对方响应与否。

缓存 Cache-Control

关于缓存的，服务器的设置比较常用的就是这些，可以通过设置缓存，允许/阻止 浏览器或者是代理服务器进行缓存提高性能，或者是考虑安全问题等等。基于缓存之上的重新验证，可以对缓存进行验证处理。

资源验证

对于服务器的资源，当浏览器此前缓存了资源，服务器的资源进行更新之后，如何判断此时的资源如何进行更新。利用验证可以进行资源的更新。

Cookie

cookie是一个比较重要的内容，虽说有许许多多的问题，但是依旧在使用，使用的过程中有许多注意的事项需要注意，如果不注意可能会cookie会被劫持或者更改，产生安全问题。

长连接 Connection

http协议是建立在TCP连接之上的，对于TCP连接是存在三次握手的过程的，假设每一次HTTP请求响应，都要重新建立TCP连接，是一个比较耗费时间和性能的事。网络的效率会非常的低下，这时候长连接的出现，可以解决这个问题，就是进行完一次HTTP请求结束，先不断开TCP连接，如果请求发送或者响应，就复用TCP连接，减少三次握手的次数。但是不同的浏览器，同时存在的TCP连接数是有限的，Chrome的TCP连接数为6.而且TCP的复用也是有限制的，只有在同一个域名下，才会复用TCP连接。在HTTP1.1中，虽然可以复用TCP连接，但是TCP连接上面的HTTP请求是串行的，就是必须一个HTTP请求结束才能进行下一个HTTP请求。
设置： Connection：keep-alive

Redirect

重定向一般分为两种类型，临时重定向（典型代表302），永久重定向（301）。当访问一个网址，如果设置为临时重定向，每次访问的时候，在服务器端进行判断，然后重定向到新的地址，浏览器每一次访问都是先访问原来的地址，然后重定向到新的地址。如果设置为302的话，虽然服务器端的处理逻辑基本一致，只是响应状态码设置为301，但是浏览器端的处理却完全不同，当浏览器是第一次访问这个地址，会先访问，然后被重定向。然后这一次的操作被缓存。下一次访问的时候，不会访问原地址，会直接访问到重定向的地址，直到用户手动清理缓存。这个是要注意的一点。可能利用不当，会产生我们不愿意看到的效果。

HTTP2/HTTPS

https应该是出现的比较早的，在之前的HTTP请求中，所有的请求都是明文传输的，没有进行加密，这种情况的话，我们利用抓包，可以看到请求的具体信息，非常的不安全。https是基于HTTP协议之上，利用公钥加密，在传输的过程中是传输的密文，到了服务器或者浏览器。再进行解密。这种方式，保证了数据的安全性，但是握手的成本加大了，更加的消耗性能。
